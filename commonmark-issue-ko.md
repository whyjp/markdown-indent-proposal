# 마크다운에 들여쓰기가 없어서, 만들어 봤습니다

마크다운이 사실상 표준이 된 지 오래입니다. Obsidian, Notion, GitHub, 위키, 채팅
도구까지. 쓰는 사람 입장에서는, 기본 서식 정도는 당연히 되리라 생각합니다.

실제로 거의 다 됩니다. 하나만 빼고 — **들여쓰기.**

시각적 깊이는 사소해 보여도, 문서를 읽는 흐름에 큰 차이를 만듭니다. 들여쓰기가
있으면 독자는 자연스럽게 포함 관계와 계층을 느낍니다. `>`와 `-`에는 인용과
목록이라는 고유한 역할이 이미 있고, 들여쓰기를 대신하기엔 맞지 않습니다.
마크다운에는 이를 위한 문법 자체가 없습니다.

---

## 우회법은 있지만, 전부 문제가 있습니다

| 방법 | 문제 |
|------|------|
| `&nbsp;` 반복 | 깊이 3이면 엔티티 12개. 소스가 엉망이 됨. |
| `>` 블록인용 중첩 | 스크린 리더가 "인용"으로 읽음 — 인용이 아닌데. 의미가 오염됨. |
| 스페이스 4칸 / 탭 | CommonMark에서 `<pre><code>` 처리. 코드 블록 용도로 이미 점유. |
| 빈 리스트 아이템 | 대부분 파서가 일반 목록으로 보정해 버림. 소용없음. |

소스를 망치거나, 의미를 왜곡하거나, 아예 안 되거나. 셋 중 하나입니다.

---

## 제안: `.>` 문법

```markdown
.> 1단계 들여쓰기.
접두사 없이 이어 쓰면 빈 줄이 나올 때까지 같은 깊이.

..> 2단계. 1단계 안에 중첩.
...> 3단계.

여기서부터 들여쓰기 밖.
```

### 렌더링 미리보기

위 문법이 파서에 반영되면 렌더링 결과는 이렇습니다:

| 소스 | 렌더링 |
|------|--------|
| `일반 텍스트.` | 일반 텍스트. |
| `.> 한 단계.` | &emsp;한 단계. |
| `이어지는 줄.` | &emsp;이어지는 줄. |
| `..> 두 단계.` | &emsp;&emsp;두 단계. |
| `...> 세 단계.` | &emsp;&emsp;&emsp;세 단계. |
| `들여쓰기 밖.` | 들여쓰기 밖. |

**점 개수가 곧 깊이입니다.** `>`(블록인용)와 같은 container block 방식 — lazy
continuation으로 이어지고, 빈 줄이면 끝. 블록인용을 쓸 줄 알면 새로 배울 게
없습니다.

그리고 `.>`, `..>`, `...>` — 소스 텍스트만 봐도 점이 늘어나는 모양에서 깊이가
읽힙니다. 렌더링 전에도 구조가 눈에 들어온다는 건, 마크다운이 원래 추구하던
바이기도 합니다.

지원하지 않는 파서에서도 그냥 일반 텍스트로 표시됩니다. 깨지는 건 없습니다.

---

## HTML로 하면 되지 않나?

`<div style="padding-left:2em">`를 쓰는 순간 마크다운이 아니게 됩니다.
터미널에서 읽을 수도, diff로 비교할 수도, 마크다운 도구로 처리할 수도 없습니다.
닫는 태그도 필요하고, 중첩하면 할수록 복잡해집니다.

마크다운의 핵심은, 렌더링 없이도 읽히는 소스입니다. HTML에 기대면 이 원칙이
무너집니다. `.>`는 이 원칙을 지킵니다.
