# 마크다운에는 들여쓰기가 없습니다. 직접 고쳐보려 합니다.

마크다운은 조용히 주류가 됐습니다. GitHub의 개발자들로 시작해 Obsidian, Notion 같은 노트 앱, 위키, 지식 베이스, 챗 도구까지 퍼졌습니다. 오늘날 마크다운을 쓰는 사람 중 상당수는 그것을 선택한 게 아니라 — 그냥 자신의 도구가 쓰는 방식이라서 씁니다.

이 변화가 중요한 이유가 있습니다. 새로운 사용자들은 다른 기대를 가지고 옵니다: 기본적인 서식은 그냥 되어야 한다는 것. 대부분은 됩니다. 제목, 굵게, 리스트, 코드 — 자연스럽게 느껴집니다. 하지만 텍스트 블록을 시각적으로 들여쓰려는 순간 — 인용도, 리스트도 아닌, 그냥 _오른쪽으로 밀고_ 싶은 것 — 마크다운에는 아무것도 없습니다.

공식적인 방법은 없습니다. 하나도.

이 벽에 부딪힌 사람들은 보통 둘 중 하나입니다. 어떻게든 억지로 우회하거나, 아니면 조용히 마크다운을 포기하거나. 이 글은 그 두 부류 모두를 위해 씁니다.

---

## 쓰지 말았어야 할 우회 방법들

오늘날 존재하는 모든 방법은 다른 무언가를 오용합니다.

### `&nbsp;` 반복

```markdown
&nbsp;&nbsp;&nbsp;&nbsp;내용
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;더 깊게
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;3단계
```

깊이 3에 `&nbsp;` 12개. 소스가 읽히지 않습니다 — 마크다운을 쓰는 이유 자체가 사라집니다.

### 블록인용 `>` 남용

```markdown
> 개요
> > 세부사항
> > > 더 깊이
```

스크린 리더는 _"인용… 인용 속 인용… 인용 속 인용 속 인용"_ 을 읽습니다 — 인용이 전혀 아닌데. 렌더링 결과가 콘텐츠의 의미에 대해 거짓말하는 것입니다. 단순히 보기 나쁜 게 아니라 시맨틱이 오염된 겁니다.

### 탭 / 스페이스 4칸

```markdown
    들여쓴 것처럼 보이지만
    사실은 코드 블록입니다
```

CommonMark에서 4칸 스페이스 = `<pre><code>`. 가장 직관적인 들여쓰기 방법이 이미 코드 블록에 점유되어 있습니다.

### 빈 리스트 아이템 트릭

```markdown
-
  들여쓴 텍스트
```

대부분의 파서가 이를 일반 리스트로 교정합니다. 렌더러마다 결과가 다릅니다. 처음 실패하고 나면 대부분 포기합니다.

## 하나의 제안: `.>`

들여쓰기에 전용 문법이 있다면 어떨까요?

```markdown
.> 이 줄에서 indent 컨테이너가 시작됩니다.
접두사 없이도 계속 안에 속합니다 (lazy continuation).
빈 줄이 올 때까지.

..> 2단계 — 1단계 안에 중첩.
...> 3단계.

빈 줄이 모든 컨테이너를 닫았습니다.
```

규칙은 단순합니다: **점의 개수 = 깊이**. `.>`는 1단계, `..>`는 2단계, 최대 `....>`.

`>` (블록인용)과 동일한 container block 모델을 따릅니다 — lazy continuation을 지원하고, 빈 줄로 종료됩니다. 사람들이 실제로 들여쓰기를 생각하는 방식과 일치합니다: 줄 단위 선언이 아닌 _영역_ 으로.

**HTML 출력:**

```html
<div class="md-indent md-indent-1">
  <p>들여쓴 내용</p>
  <div class="md-indent md-indent-2">
    <p>더 깊이</p>
  </div>
</div>
```

권장 CSS: `padding-left: calc(N × 1.5em)`
접근성: `role="group" aria-level="{N}"`

## 그냥 HTML을 쓰면 안 될까요?

마크다운은 이미 인라인 HTML을 허용합니다. 그러면 `<div style="padding-left:2em">`를 쓰면 되지 않을까요?

```markdown
<div style="padding-left:2em">
  내용
  <div style="padding-left:2em">
    더 깊이
  </div>
</div>
```

그건 더 이상 마크다운이 아닙니다.

소스를 사람이 읽을 수 없게 됩니다. 터미널에서 읽을 수도, diff를 깔끔하게 볼 수도, 마크다운을 기대하는 도구로 처리할 수도 없습니다. 닫는 태그, 인라인 스타일, 깊이가 늘어날수록 복잡해지는 중첩이 필요합니다.

마크다운의 핵심은 소스 자체가 문서라는 것입니다 — 렌더링 없이도 읽을 수 있어야 합니다. HTML fallback은 쓰는 사람, 원본 파일을 읽는 사람, 그 사이의 모든 도구에 대한 그 약속을 깹니다.

`.>`는 그 약속을 지킵니다.

## 설계 결정

**기존 문법과 무충돌.**
줄 시작의 `.>`는 현재 어떤 CommonMark 또는 GFM 파서에서도 특별한 의미가 없습니다. 진정으로 비어있는 공간입니다.

**점진적 호환 (Graceful degradation).**
미지원 파서에서는 `.>`가 일반 텍스트로 표시됩니다. 내용은 읽을 수 있고, 시각적 들여쓰기만 사라집니다. 지금 당장 써도 아무것도 깨지지 않습니다.

**타이핑 편의성.**
`.`과 `>`는 같은 물리 키 시퀀스를 씁니다 (`.` → `Shift+.`). 더 깊은 들여쓰기 = `.` 한 번 더. 손이 키에서 떠나지 않습니다.

**역방향 중첩 불가.**
`.>`는 column 0에서만 컨테이너를 엽니다. `>`, `-`, 테이블 셀, 코드 펜스 내부에서는 사용할 수 없습니다. 파싱이 명확하고 예측 가능해집니다.

**최대 깊이 4 권장.**
`....>`를 넘어선다면 그건 거의 확실하게 문법 문제가 아니라 콘텐츠 설계 문제입니다.

## 구현 명세 초안

1. `/^\.{1,N}> /` 패턴으로 시작하는 줄은 깊이 N의 indent container를 엽니다
2. Lazy continuation: 접두사 없는 다음 줄은 현재 컨테이너에 속합니다
3. 빈 줄이 컨테이너를 종료합니다
4. 더 깊은 `.>` (예: `.>` → `..>`)는 중첩 컨테이너를 만듭니다
5. 모든 인라인 마크다운이 내부에서 작동합니다
6. 내부 블록 (blockquote, list, fenced code)은 정방향 합성으로 허용됩니다
7. 다른 블록 내부 (`>`, `-`, 테이블 셀, 코드 펜스)에서는 인식되지 않습니다 — column 0 전용
8. HTML 출력: `<div class="md-indent md-indent-{N}">` with `role="group" aria-level="{N}"`
