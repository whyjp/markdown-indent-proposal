# CommonMark와 충돌하지 않는 네이티브 들여쓰기 문법: `.>` 제안

## Use case or problem

옵시디언의 Tab 액션과 4칸 들여쓰기는 **많은 사용자가 원하는 그대로** 동작합니다.
미리보기에서 indent와 layer 그룹으로 잘 보여주고요. 기능 자체는 매우 훌륭합니다.

문제는 **4칸 공백이 CommonMark와 충돌한다는 점**입니다. 스펙에서는 4칸이
*코드 블록*을 의미합니다. 옵시디언은 *들여쓰기*로 해석합니다. 그래서 옵시디언
들여쓰기에 의존한 노트는 밖으로 나가는 순간 호환성이 깨집니다.

| 맥락 | 4칸 공백 / 탭의 의미 |
|------|----------------------|
| **CommonMark 스펙** | 코드 블록 (`<pre><code>`) |
| **옵시디언** | 들여쓰기, layer 그룹 |

옵시디언이 구현한 기능은 매우 훌륭합니다. 다만 아쉽게도, 마크다운 저변을
넓힌 것도 맞지만 스펙과 어긋난 것도 맞습니다. 이로 인해 두 가지 문제가 생깁니다:

1. **서로 다른 mental model** — "4칸 = 코드 블록"은 스펙에서 기본 규칙입니다.
   Tab/4칸으로 들여쓰는 옵시디언 사용자는 다른 규칙을 배우게 됩니다. 노트를
   공유하거나 다른 도구로 옮기면 그 차이가 드러납니다.

2. **호환성이 깨짐** — 같은 `.md` 파일을 GitHub, VSCode 미리보기, Obsidian Publish,
   또는 다른 CommonMark 렌더러에서 보면, 들여쓴 문단이 코드 블록으로 표시됩니다.
   노트가 옵시디언 밖에서는 조용히 다른 의미로 바뀝니다.

옵시디언 볼트는 일반 `.md` 파일입니다. 노트는 GitHub, GitLab, VSCode 미리보기,
Obsidian Publish 등에서 공유됩니다. 4칸 들여쓰기에 의존하는 사용자는 *옵시디언
안에서는* 훌륭한 경험을 하지만, 노트는 더 이상 이식 가능하지 않습니다. 사용자에게
선택을 강요하는 셈입니다: 들여쓰기를 쓰면 이식성이 깨지고, 스펙을 따르면 indent
UX를 포기해야 합니다.

**사용 사례: 번호 없이 종속 관계만 표현** — 어떤 문단은 다른 문단 아래에 속해
있습니다. 문맥상 종속이지만 목록 항목이 아니고, 인용도 아니며, 별도 헤딩이
필요하지 않습니다. 헤딩과 목록이 많은 경우를 커버하지만, *시각적 종속만 있는
흐름형 문단*은 다루지 못합니다.

## Proposed solution

`.>`를 네이티브 들여쓰기 문법으로 도입 — 같은 indent + layer UX를 유지하면서
CommonMark와 충돌하지 않음.

### 문법

```markdown
.> 1단계 들여쓰기.
접두사 없이 이어 쓰면 빈 줄이 나올 때까지 같은 깊이.

..> 2단계. 1단계 안에 중첩.
...> 3단계.

여기서부터 들여쓰기 밖.
```

### 규칙

- **점 개수 = 깊이.** `.>` = 1, `..>` = 2, `...>` = 3 등.
- `>`(인용)과 같은 방식: lazy continuation, 빈 줄로 종료.
- `<div class="md-indent md-indent-N">`로 렌더링 — CSS로 스타일링 용이.
- 미지원 파서에서는 일반 텍스트로 표시. 깨지지 않음.

### 렌더링 미리보기

| 소스 | 렌더링 |
|------|--------|
| `일반 텍스트.` | 일반 텍스트. |
| `.> 한 단계.` | 한 단계 들여쓰기. |
| `이어지는 줄.` | 들여쓰기 이어짐. |
| `..> 두 단계.` | 두 단계 들여쓰기. |
| `들여쓰기 밖.` | 들여쓰기 밖. |

### 옵시디언에 맞는 이유

1. **같은 UX, 충돌 없음** — indent + layer 그룹을 유지하면서, CommonMark가
   예약하지 않은 문법 사용. "4칸 = 코드 블록"을 덮어쓸 필요 없음.
2. **이식성** — 노트가 GitHub, VSCode, Publish, 지원하는 모든 렌더러에서
   동작. 조용히 깨지지 않음.
3. **스펙 정렬** — 스펙과 어긋나지 않음. 사용자가 도구를 가리지 않고 하나의
   mental model을 유지할 수 있음.
4. **소스 가독성** — `.>`, `..>`, `...>`로 원문만 봐도 깊이가 보임. HTML 불필요.
5. **폴딩 가능** — 들여쓰기 블록은 목록 중첩처럼 폴드 영역으로 자연스럽게 매핑.
6. **공유 스펙·공유 구현** — [markdown-it-dot-indent](https://www.npmjs.com/package/markdown-it-dot-indent)(npm)로 실효성 검증 중 — npm 배포, VSCode 확장, 사용 테스트. 옵시디언이 이 규칙을 채택하면 같은 패키지나 스펙을 공유해 생태계 전반의 업데이트·유지보수를 맞출 수 있습니다.
7. **안전한 폴백** — 미지원 파서에서는 일반 텍스트. 노트는 그대로 유효.

## Current workaround (optional)

- **Tab / 4칸 들여쓰기** — 옵시디언에서는 동작(indent + layer 그룹). 하지만
  이식성이 깨짐. 같은 파일이 GitHub, VSCode 등에서는 코드 블록으로 표시됨.
- **`>` 인용 중첩** — 의미 오염(스크린 리더가 "인용"으로 읽음); 인용이 아닌
  콘텐츠에는 부적합.
- **불릿 + CSS로 숨기기** — 불안정하고 테마에 의존함.
